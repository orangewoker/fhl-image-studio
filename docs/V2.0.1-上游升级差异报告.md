## 上游主要变化分类

### 1. 新增 Gio 桌面客户端

上游新增完整 `gio-client/`，并把 `go.work` 扩展为同时包含 `gio-client`、`go-cli`、`image-studio`、`shared/compat-go`。

价值:
- 这是一个独立 Go/Gio 桌面端，目标是减少对 WebView/Wails 的依赖。
- UI 大量对齐 Windows Fluent 布局，并带自己的历史、结果详情、raw response、批量结果网格。

对 V2.0.1 的建议:
- 暂不合入。你的 V2.0.1 当前主线是 `start-ui.cmd + Vite 浏览器模式 + 本地 CLI/Skill`，而且根目录已经做了小白友好整理。
- 若未来想做“无浏览器依赖桌面版”，再单独开分支评估 Gio，不要混进这次 2.0.1 小升级。

### 2. 跨端兼容状态桥

上游新增 `shared/compat-go`、`image-studio/backend/compatibility.go`、前端 `compatState.ts`，用于在 WebView2/Gio 等客户端之间同步 profiles、历史、设置、输出目录等状态。

价值:
- 多客户端之间共享历史和配置更方便。

风险:
- 上游实现会读写全局兼容状态和 `gptcodex.*` 风格数据。你的 V2.0.1 刚修过 API Key 泄露和浏览器缓存串库问题，已经要求 storage namespace 隔离。
- 如果直接合入，可能再次把旧 profile、旧历史、旧 API 相关槽位带进新发行版。

对 V2.0.1 的建议:
- 暂不直接合入。
- 如果未来要合，必须先改成命名空间隔离，并明确禁止兼容状态携带 API Key 或旧 keyring 槽位。

### 3. 自定义比例功能

上游新增 `CustomAspectRatioModal`、`customAspectRatios.ts`，把 `SizeValue` 扩展为任意 `${number}x${number}`，支持用户保存自定义比例。

价值:
- 对普通用户更灵活。
- 上游自动按 1K/2K/4K 估算像素尺寸。

风险:
- 你的 V2.0.1 已按 FHL 实测稳定性固定为 14 个比例，并改成下拉选择加真实比例示意图。
- 直接合入会上翻现有 UI 和测试，可能破坏 FHL 已验证比例映射。

对 V2.0.1 的建议:
- 暂不合入上游自定义比例 UI。
- 可只参考上游 `SizeValue = "auto" | \`${number}x${number}\`` 的类型放宽思路，前提是不改掉当前 14 比例列表。

### 4. 循环出图

上游新增 `LoopGenerationSection`，支持总张数、并发、自动另存为、关闭实时预览等；对应 store 增加 loop controller、`disablePreview` 和 save-to-dir 能力。

价值:
- 适合批量跑图。
- `disablePreview` 能降低大量任务时的内存压力。

风险:
- 你的 V2.0.1 已有浏览器后台任务代理、最近任务流、失败槽位、顺序执行、多参考图稳定策略。
- 上游循环出图是前端 store 内控制任务，直接合入可能绕开或冲突后台任务代理，导致刷新续跑、失败保留、任务流顺序出问题。

对 V2.0.1 的建议:
- 暂不直接合入。
- 后续如果要做 V2.1，可把“循环出图”设计为后台代理的 task group 扩展，而不是复制上游前端循环 controller。

### 5. NewAPI / Images API 兼容

上游新增 `imagesNewAPICompat`：
- profile 增加开关。
- Images API 请求强制 `response_format=b64_json`。
- 开启后不发送 `stream` / `partial_images`。
- Responses/Image payload 增加 `disablePreview`。

价值:
- 对 NewAPI / Packy 风格中转站的 Images API 兼容更好。

风险:
- 你的 FHL 图生图已经验证为 Responses + SSE 最稳定，不能被这个开关带回 Images API 默认路径。
- FHL 默认 profile 必须继续是 `responses / openai / https://www.fhl.mom / gpt-5.5 / gpt-image-2 / partial_images=1`。

对 V2.0.1 的建议:
- 可以作为“非默认备用能力”择机合入：只在用户手动选择 Images API profile 时显示兼容开关。
- 不允许影响 FHL 一键配置和图生图默认路径。

### 6. 历史加载与预览性能

上游增加分页加载历史、历史缩略图加载 fallback、批量 persist history、导入历史时更谨慎处理本地路径。

价值:
- 历史很多时启动更快。
- 缩略图找不到时 UI 更稳。

风险:
- 需要适配 V2.0.1 的 IndexedDB namespace、根目录输出路径、任务流补历史逻辑。

对 V2.0.1 的建议:
- 值得优先挑选合入，但必须小步移植。
- 先合缩略图 fallback 和分页读取，再跑历史/结果流回归。

### 7. 保存提示与自动另存为

上游新增生成后“是否另存”弹窗，后端增加 `SaveImageToDir` / `SaveImagePathToDir`。

价值:
- 普通用户更容易把结果保存到目标目录。
- 可为未来批量自动保存提供基础。

风险:
- V2.0.1 已规定根目录 `output` / `intermediate` 的语义，另存提示如果太频繁会打扰主流程。

对 V2.0.1 的建议:
- 保存到指定目录能力可合入。
- 生成后弹窗默认不建议开启，除非用户明确要这个交互。

### 8. Android / Mac / Windows UI 细节

上游有 Android 提交导航、进度布局、Mac 上游弹窗、Windows/Gio 视觉对齐、图标/截图/README 更新。

价值:
- 如果继续维护 Android 壳，这部分值得逐条看。

风险:
- 多数是视觉和平台端 UI 细调，对 FHL 浏览器发行包优先级不高。

对 V2.0.1 的建议:
- 只挑 Android 明确 bugfix。
- 不合 README/截图/品牌覆盖，避免冲掉 FHL 自己的发行说明。

## 与 FHL 魔改高冲突的点

- 上游 profile 默认不是 FHL；不能覆盖 `ensureFHLResponsesProfile`。
- 上游兼容状态会同步 profiles/history/settings；不能绕过 `storageNamespace` 和 `keyringUserFor(profileId)` 命名空间隔离。
- 上游自定义比例会重写 `sizeCapabilities.ts`；不能覆盖你已实测的 14 比例映射。
- 上游循环出图前端 controller 与你的浏览器后台任务代理/最近任务流冲突。
- 上游 Images API 兼容开关只能作为手动备用，不能改变 FHL 默认 Responses 图生图。
- 上游 docs/brand/release workflow 不能覆盖你独立仓库、Skill、根目录极简发布结构。

   - 历史缩略图 fallback。
   - 历史分页加载思路，适配命名空间后再合。

   - 合入 `imagesNewAPICompat` 字段和 UI 开关。
   - 只对 Images API profile 生效。

3. 保存能力按需合入:
   - 可合 `SaveImagePathToDir` / `SaveImageToDir`。
   - 不默认弹保存提示。

4. 暂缓:
   - Gio 客户端。
   - 上游 compat state 全量同步。
   - 上游循环出图。
   - 上游自定义比例 UI。
